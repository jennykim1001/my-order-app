[서비스 이름] 핫한 디저트 가게 "오늘의 주문" 서비스

[한 줄 소개]
줄 서지 말고, 담아두고 오세요
오늘 판매하는 메뉴를 보여주고, 회원가입한 손님이 메뉴를 담아 주문하면
손님은 "내 주문"에서 자기 주문 상태를, 사장은 관리 화면에서 전체 주문을 확인하는 서비스

[핵심 기능 — 6가지]
1. 메뉴 보여주기: 오늘의 메뉴 카드 목록 (로그인 없이 누구나 볼 수 있음)
2. 회원가입·로그인: 손님이 이메일로 가입하고 로그인한다 (Supabase Auth)
3. 주문 담기 & 접수: 로그인한 손님이 메뉴를 담아 주문을 저장한다 (Create)
4. 내 주문 보기: 손님이 자기가 남긴 주문과 상태만 본다 (Read — 자기 것만!)
5. 사장 전체 조회 & 완료 처리: 사장만 전체 주문을 보고 "접수 → 완료"로 바꾼다 (Read/Update)
6. 주문 삭제: 사장이 장난·취소 주문을 삭제한다 (Delete)

[화면 구성 — 경로(라우트) 기준]
■ 주문 페이지 (경로 "/" — src/pages/OrderPage.jsx)
- 상단: 가게 이름 + 장바구니 배지 + (비로그인) [로그인] / (로그인) [내 주문] [로그아웃]
- 중간: 오늘의 메뉴 카드 목록 — MenuCard 컴포넌트 반복
- 하단: 주문서 — 담은 품목(수량 +/−), 합계, 픽업 희망 시간, [주문하기]
        (비로그인 상태면 [주문하기] 대신 [로그인하고 주문하기])

■ 회원가입·로그인 (경로 "/auth" — src/pages/AuthPage.jsx)
- shadcn Tabs로 [로그인] / [회원가입] 전환 (이메일, 비밀번호)

■ 내 주문 (경로 "/my" — src/pages/MyOrdersPage.jsx)
- 로그인한 손님의 주문만 최신순 목록 (품목, 금액, 상태 Badge, 주문 시각)

■ 관리 화면 (경로 "/admin" — src/pages/AdminPage.jsx)  
- 사장 로그인 → 전체 주문 표 + 각 행에 [완료] [삭제] 버튼 + 로그아웃

[데이터 — 저장할 것]
- 주문자(user_id — 로그인한 계정과 자동 연결, RLS가 위조 방지) / 픽업 희망 시간 (텍스트)
- 주문 내역 (텍스트 — 예: "크루아상 x2, 쑥라떼 x1") / 합계 금액 (숫자)
- 상태 (텍스트, 기본값 "접수") / 주문 시각 (자동)
- profiles 테이블: id(=auth 계정), role (텍스트, 기본값 "customer", 사장 계정만 수동으로 "owner"로 변경)

[메뉴 데이터 — DB가 아니라 코드에 둔다]
- 오늘의 메뉴는 src/data/menu.js 배열 (id, 이름, 가격, 설명)
- 메뉴 등록/수정 기능은 만들지 않는다 — 사장이 파일을 직접 고친다

[회원·권한 설계]
| 역할 | 가입 방법 | 할 수 있는 일 |
|---|---|---|
| 방문자(비로그인) | — | 메뉴 보기만 |
| 손님(로그인) | 회원가입 화면에서 직접 | 주문 남기기(자기 명의로만) + "자기" 주문만 조회. 상태 변경·삭제는 불가 |
| 사장(로그인) | 계정은 손님과 같은 방식, `profiles.role = 'owner'`로 구분 | 전체 조회·완료 처리·삭제 |

- 사장 판별 근거: `profiles` 테이블에 `role` 컬럼(기본값 `'customer'`)을 두고, 특정 계정에만 `'owner'`를 수동으로 지정한다 (Supabase 대시보드에서 SQL로 직접 UPDATE — 코드에 이메일을 하드코딩하지 않는다).
- RLS 정책은 반드시 이 `role` 컬럼을 참조해서 작성한다:
  - orders SELECT: `user_id = auth.uid() OR (select role from profiles where id = auth.uid()) = 'owner'`
  - orders INSERT: `WITH CHECK (user_id = auth.uid())` — 클라이언트가 다른 사람의 user_id를 넣어 주문을 대신 남기는 것을 서버(RLS)가 원천 차단한다.
  - orders UPDATE/DELETE: `(select role from profiles where id = auth.uid()) = 'owner'`인 경우만 허용 — 손님 계정은 자기 주문이라도 수정·삭제 불가.
- `/admin` 라우트 가드(프론트)는 UX 편의일 뿐 보안 경계가 아니다. 손님 계정이 URL을 직접 입력해 `/admin`에 접근해도, 실제 데이터 조회·변경은 위 RLS가 막는다 — "화면을 안 보여준다"가 아니라 "요청 자체가 거부된다"가 진짜 방어선이다.
※ 손님 A는 손님 B의 주문을 절대 볼 수 없다 (화면이 아니라 RLS가 보장)
※ "사장 가입 버튼"은 없다 — 사장 지정은 화면 밖(Supabase)에서, role 컬럼 UPDATE로만 이뤄진다
